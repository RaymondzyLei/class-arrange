# 优化选课方案计算设计

## 目标

在不改变现有排课结果、排序规则和默认行为的前提下，解决选课或偏好变化时主线程同步生成完整笛卡尔积造成的明显卡顿，并补齐两类交互：

- 排课支持“自动排课”和“手动开始排课”。
- 所有课程选择入口同时支持选择/移除当前时间组与该课程的所有时间组。

旧用户默认保持自动排课。手动模式下，编辑课程或偏好不会清空或替换当前课表；页面继续展示上一次成功计算的结果，并明确显示“待重新计算”和“开始排课”。

## 已确认需求

- 计算模式属于排课自定义设置，取值为 `auto` 或 `manual`。
- “自定义”弹窗和首次使用引导均可选择计算模式。
- 自动模式延续现有体验：课程或偏好变化后自动计算，但计算期间不阻塞交互，并保留旧课表直到新结果完整生成。
- 手动模式中，课程、早八倾向、空半天倾向、不可排时间等输入变化后：
  - 保留上一次成功计算的课表和方案列表；
  - 标记为“待重新计算”；
  - 显示“开始排课”或“重新计算”按钮；
  - 点击后显示加载状态；
  - 新结果成功后一次性替换旧结果；失败或任务过期时继续保留旧结果。
- 切换排课方案或学期时必须硬重置计算快照，不能短暂展示另一个方案/学期的旧课表。
- 计算结果必须与当前算法精确一致，包括方案 ID、前 8 个方案的顺序、冲突数、课程数、总学分和总学时。
- 所有课程选择入口提供四种语义一致的操作：
  - 选择当前时间组；
  - 选择该课程全部时间组；
  - 移除当前时间组；
  - 移除该课程全部时间组。

## 当前瓶颈

`src/utils/arrangement.ts` 当前会：

1. 将同一课程的多个已选时间组作为候选桶；
2. 递归构造并保留完整笛卡尔积；
3. 对每个组合重新展开周次、扫描所有时间槽并计算冲突与偏好；
4. 生成全部 `Arrangement`；
5. 对全部结果排序后只保留前 8 个。

组合数随候选课程指数增长；大量短命数组、集合、完整排序和 React 主线程上的同步计算共同造成卡顿。

## 交互状态模型

新增 `useArrangementCalculation` 作为唯一计算编排入口，将“正在编辑的输入”与“已应用到课表的快照”分离。

```ts
type CalculationMode = 'auto' | 'manual';
type CalculationPhase = 'empty' | 'dirty' | 'calculating' | 'ready' | 'error';

interface AppliedCalculationSnapshot {
  inputRevision: number;
  arrangements: Arrangement[];
  selectedArrangementId: string | null;
  settings: CustomScheduleSettings;
}
```

状态转换：

```text
无已选课程 -> empty
empty + 有输入 -> auto: calculating / manual: dirty
ready + 输入变化 -> auto: calculating / manual: dirty
dirty + 点击开始排课 -> calculating
calculating + 成功且任务仍为最新 -> ready（原子提交完整快照）
calculating + 失败 -> error（保留旧快照）
calculating + 又有新输入 -> 取消旧任务；auto 启动新任务，manual 回到 dirty
方案或学期切换 -> empty（硬重置）
```

显示规则：

- `dirty` 且存在旧快照：课表和方案面板显示旧快照，并显示“待重新计算 / 重新计算”。
- `dirty` 且不存在旧快照：显示空课表提示和“开始排课”。
- `calculating`：按钮禁用并显示加载动画；旧快照存在时继续展示。
- `error`：显示非阻塞错误和重试按钮，旧快照不丢失。
- 课表冲突高亮、统计和不可排时间均读取已应用快照，而不是尚未计算的编辑输入。

每次启动任务生成单调递增的 generation token。只有 token 仍为最新的任务可以提交结果，以避免快速修改输入后旧结果覆盖新结果。

## 精确 Top 8 计算引擎

### 保持现有排序契约

比较顺序必须与现有实现完全一致：

1. 冲突课程组数量升序；
2. 启用“优先空出半天”时，空半天得分降序；
3. 启用“优先减少早八”时，早八天数升序；
4. 排序后的 group key 字符串字典序升序；
5. 总学分降序。

现有最后一项通常不会参与决定，因为不同组合的 group key 已不同，但仍保留以保证契约不变。

### 预计算

每个 `CourseGroup` 只计算一次：

- 展开周次后的时间槽 key；
- 工作日 × 节次占用位图；
- 早八日期位图；
- 是否命中不可排时间；
- 学分和学时；
- 与其他 group 的冲突邻接表。

冲突邻接表通过“时间槽 -> group 列表”的倒排索引构建，避免组合内重复比较所有课程。

### 深度优先搜索

- 唯一时间组课程先作为固定集合加入累积状态。
- 按课程候选桶深度优先选择一个时间组，不再构造完整笛卡尔积数组。
- 递归状态增量维护：
  - 每个 group 的冲突引用计数；
  - 当前冲突 group 数；
  - 每天节次占用计数/位图；
  - 早八日期位图；
  - 总学分和总学时；
  - 当前 group key 列表。
- 到达叶子时只构造一个候选摘要。
- 使用容量为 8 的最大堆维护当前最优结果；叶子候选优于堆顶时替换。
- 搜索完成后仅对最多 8 个结果做最终排序并生成 `Arrangement[]`。

允许加入严格安全的分支界限：只有剩余候选无论如何都不可能优于当前第 8 名时才剪枝。无法证明时继续搜索，绝不以启发式剪枝牺牲精确性。最坏情况仍可能遍历完整组合数，但内存从完整笛卡尔积规模降为 `O(搜索深度 + 图 + 8)`，并消除每个组合的重复全量扫描和全量排序。

## Worker 边界

精确搜索放入 Web Worker，避免 CPU 密集任务阻塞 React 主线程。主线程只发送可结构化克隆的最小 DTO：

- group key、course code、展开后的时间槽/位图；
- 学分、学时；
-计算偏好和不可排时间；
- generation token。

Worker 返回前 8 个方案的 group key、指标和 token；主线程使用现有 group 索引还原 `Arrangement.groups`。输入变化时终止旧 Worker 或忽略旧 token。若构建环境不支持 Worker，提供相同接口的同步回退，便于测试和非浏览器环境运行。

## 课程选择操作

建立共享选择工具，避免各入口自行拼接 ID：

```ts
idsForGroup(group: CourseGroup): string[]
idsForCourse(courseCode: string, groupsByCode: Map<string, CourseGroup[]>): string[]
```

计划选择的课程 ID 是编辑输入；手动模式下编辑后仅令计算状态变为 `dirty`。以下入口统一接入：

- 课程池条目；
- 课程详情弹窗；
- 已选课程管理；
- 培养方案中的单个候选和候选选择弹窗；
- 一键添加必修课程流程。

按钮文案必须显式区分“这个时间组”和“全部时间组”，并根据当前选择状态切换为对应的移除动作。数组批量增删继续通过 reducer 完成，去重且保持确定性顺序。

## 设置与兼容

`CustomScheduleSettings` 增加：

```ts
calculationMode: 'auto' | 'manual';
```

- 默认值为 `auto`，旧版 localStorage 缺少字段时迁移为 `auto`。
- 自定义弹窗将模式作为单选/分段选项展示，文案为“自动排课”和“手动开始排课”。
- 首次使用引导在基础偏好步骤加入相同选择和简短说明。
- 模式本身可以立即改变计算编排方式，但不会无故丢弃最近成功快照。

## 正确性验证

保留当前实现作为仅测试使用的暴力枚举 oracle，生产代码改用精确 Top 8 引擎。测试包括：

- 固定样例覆盖所有排序键和不可排时间；
- 随机生成小型课程桶，将优化引擎与 oracle 的方案 ID、顺序和全部指标逐项比较；
- 自动/手动状态机测试：保留旧快照、dirty、加载、失败、过期结果、方案/学期硬重置；
- 课程选择工具测试：当前时间组/整门课程的添加、移除和幂等性；
- 组件交互测试：自定义、引导、课程池、详情和培养方案入口文案与回调。

性能测试使用确定性的大候选桶，只设置宽松的回归上限并验证不会创建完整组合数组；性能数字不替代结果一致性测试。

## 非目标

- 不改变方案排序偏好或将近似结果冒充精确结果。
- 不把排课结果上传到服务器。
- 不在本分支提交 2026 秋/夏两份大型课程数据文件；该数据由 `supplement-course-information` 分支独立维护。
- 不在本轮重新设计课表视觉或培养方案数据模型。
